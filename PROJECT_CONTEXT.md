## 关键结论
- SeatWise Campus 是 **C/S 架构**自习室预约系统，两大工程 `client` / `server`，`docs` 为跨端通用文档。
- 系统的技术核心只有两个：**时间片并发选座的正确性** 与 **多客户端实时看板的一致性**。其余功能围绕它们展开。
- 正确性来源永远是 **MySQL**；Redis 只做加速与调度。任何设计不得违反此点。

## 一、项目背景
高校自习室存在“空位不透明、线下占座、到场无座、使用率难统计、状态更新滞后、爽约不公平”等痛点。SeatWise Campus 面向**多校区、多楼栋、多楼层、多自习室**场景，提供在线选座、限时预约、签到、超时释放、爽约黑名单、实时热力图、统计报表，并预留积分排名与附近空位推荐。

## 二、用户角色
| 角色 | 英文 | 能做 | 不能做 |
| --- | --- | --- | --- |
| 学生 | STUDENT | 登录、筛选、查看空位、预约、签到、取消、查看记录/积分/排名/附近空位 | 维护基础数据 |
| 管理员 | ADMIN | 维护校区/楼栋/楼层/自习室/座位排布、看板、报表、黑名单、积分记录 | 冒用学生身份操作他人预约 |
| 黑名单用户 | STUDENT(受限) | 登录、查看历史记录 | 在限制期内预约 |
| 系统(定时任务) | SYSTEM | 超时释放、统计聚合、积分结算 | — |

## 三、核心业务流程（概览）
登录 → 按校区/楼栋/楼层/自习室筛选 → 查看座位状态 → 选日期与时间片 → 网格选座 → 提交预约（并发控制）→ 待签到 → 到场签到 → 使用 → 签退完成；或超时未签到 → 自动释放 + 爽约计数（可能进黑名单 + 扣积分）。看板全程 SSE 实时更新。详见 [docs/03-core-business-flows.md](docs/03-core-business-flows.md)。

## 四、核心技术难点
| 难点 | 说明 | 归属文档 |
| --- | --- | --- |
| 并发抢座 | 多人同一座位同一时间片，只能一人成功 | server/05 |
| 时间片建模 | 30 分钟片 + 唯一索引兜底 | server/02、server/05 |
| 超时释放 | 精确到分钟的延迟任务，避免全表扫描 | server/06 |
| 实时一致性 | 快照 + SSE 增量、断线重连、多端同步 | server/07、client/04 |
| 报表性能 | 聚合表而非实时全表扫描 | server/08 |
| 附近推荐 | 距离 + 空位 + 开放状态排序 | server/10 |

## 五、MVP 边界
**MVP 内**：登录、学生预约、自习室基础管理、座位排布、并发防重复、签到、超时释放、黑名单、实时热力图、基础报表。
**MVP+**：积分排名、最近空位推荐。
详见 [docs/04-mvp-scope.md](docs/04-mvp-scope.md)。

## 六、非目标范围（当前不做）
- 不做真实 GPS / 室内定位强依赖（附近推荐先用手动选位置 + 配置坐标）。
- 不做 AI 模型训练与推理落地（仅设计接口与文档）。
- 不做真实短信/推送通道（通知仅预留表与页面）。
- 不做支付、门禁硬件对接、跨校数据互通。
- 不做移动端原生 App / 小程序（后续扩展）。

## 七、术语约定（速览，完整见 GLOSSARY.md）
StudyRoom 自习室 / Seat 座位 / Reservation 预约 / ReservationSlot 时间片占用 / BlacklistRecord 黑名单 / Score 积分 / NearestAvailableRoom 附近空位推荐。文档中文，代码/表名/接口/枚举英文。

## 八、设计优先级
1. **正确性 > 性能 > 体验**：并发正确性最高，不得为性能牺牲。
2. **后端权威 > 前端体验**：前端校验只为体验。
3. **MVP > MVP+ > 扩展**：分级实现，禁止一次性铺开。
4. **可运行 > 完备**：小步可运行优先。
