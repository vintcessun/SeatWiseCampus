## Q1-签到时间规则缺陷

### 问题描述
1. 签到窗口规则不清晰。例如预约 9:30-10:00，在 9:55 签到时系统判定爽约。
   - 当前规则：签到窗口为预约开始后 **15 分钟内**（9:30-9:45），超出即算超时。
   - 学生端无任何提示说明签到窗口的存在与起止时间，容易误以为「只要预约时段内都能签到」。
2. 学生可以在任意时间签到任意时段的预约。例如上午 10:00 签到下午 14:00 的预约。
   - 这是不合理的——签到窗口应当基于预约开始时间动态计算，不允许提前跨时段签到。
   - 当前缺乏签到窗口的前端展示与后端越界校验。

### 待修复项
- [ ] 学生端预约详情/预约卡片上明确展示**签到窗口起止时间**（如：签到开放时间 9:30-9:45）
- [ ] 前端校验：当前时间未进入签到窗口时禁止签到操作（按钮置灰+提示）
- [ ] 后端确认：`POST /api/reservations/{id}/check-in` 是否已校验当前时间是否在签到窗口内？若当前时间早于预约开始时间或晚于签到截止时间，应返回明确的错误码（目前可能仅校验超时未签到，缺少「来得太早」的拦截）
- [ ] 增加错误码（如需）：`SIGN_IN_TOO_EARLY`（当前时间早于签到窗口）或统一归入 `INVALID_SIGN_IN_TIME`

## Q2-我的积分不与排行榜同步刷新

### 问题描述
- 右上角「我的积分」显示的是静态/初始值，积分排行榜刷新后它不随之更新。
- 积分更新后需要手动刷新页面才能看到新值。

### 待修复项
- [ ] 确认积分存储与读取路径：`sys_user.credit_score` 更新后，前端「我的积分」是否通过实时接口（`GET /api/scores/me`）读取而非本地缓存？
- [ ] 确认排行榜刷新是否触发积分重新计算，以及「我的积分」是否监听此更新
- [ ] 修复方案：积分变更（签退/取消/超时）后 SSE 推送积分变更事件，或「我的积分」组件定时/路由激活时重新拉取

## Q3-管理端不能追踪具体学生预约情况

### 问题描述
- 管理端缺乏按学生查看预约历史/当前状态的入口。
- 目前仅能看到自习室维度的报表与座位占用，无法定位到具体学生的预约记录。

### 待修复项
- [ ] 确认是否存在 `GET /api/admin/reservations?userId=&status=&date=` 接口
- [ ] 如无此接口，需新增：按 userId / 姓名 / 学号筛选预约记录，支持分页
- [ ] 管理端需新增页面入口（如学生管理→预约记录），或直接复用预约记录表增加搜索筛选
- [ ] 响应应包含：用户名、自习室、座位号、日期、时间片、状态、签到时间、签退时间
