## 基础功能缺陷

**学生端**： 

1. 目前允许一直预约，但是实际上不可能允许学生提前占用过多的未来想坐的座位 

2. 预约界面右侧未显示自己待签到/已签到的座位，或者在预约/使用中的座位上显示座位的使用时间段，否则需要点进“我的预约”里看，比较麻烦 

3. 积分排行实时更新，但是界面右上角积分没有实时更新 

4. 扣除积分/增加积分时要提示扣除/增加多少积分 

5. 预约座位开始时间会早于当前时间，如果没签到但释放，会直接扣除积分，这不合理 

**对于管理员端**：

1. 管理控制台先选时间才能看到座位情况/座位启用情况，不便捷；应该有一个实时看版，如果进一步选择时间才看那时候的座位看版

## 建议

**目标**：必须把分布式系统的动态性、实时性在前端展示出来

### 一、最值得做的核心创新：时空座位地图

#### 1. 从二维座位图升级成“时空座位图”

普通二维座位图只能表达某一时刻：

- 绿色：空闲
- 黄色：已预约
- 红色：使用中
- 灰色：不可用

可以增加一条时间轴：

```
14:00 ───── 15:00 ───── 16:00 ───── 17:00
```

用户拖动时间轴或选择时间段时，座位图动态变化。==我建议直接以时间轴当前选择的时间点作为预约的开始时间，结束时间再拖动列表元素进行选择。目前系统是以30min为时间间隔的，所以时间轴只能停留在10:00/10:30/11:00这些时间点上，中间座位的颜色使用渐变/插值==

点击座位后，不只是显示当前状态，而是显示未来占用情况：

```
A05
14:00—15:30  空闲
15:30—16:00  已预约
16:00—18:00  空闲
```

这样系统可以回答一个普通预约系统回答不了的问题：

> 哪个座位不仅现在空闲，而且能让我连续坐两个小时？

这是“空间 + 时间”联合查询，比单纯实时热力图更有功能价值。

#### 2. 直接在座位图上显示“我的预约”

你已经发现当前用户必须返回“我的预约”页面才能确认座位。

可以在座位图中增加：

- 呼吸光环：当前正在使用
- 座位角标：`15:30—17:00`
- 页面顶部固定卡片：`你将在 18 分钟后使用 B07`
- 点击右侧的预约栏的一个卡片，会自动定位并闪烁对应座位

这不仅修复缺陷，也让整个页面更像实时系统。

#### 3. 推荐座位使用动画解释，而不是直接给结果

系统推荐座位时：

- 第一推荐座位：绿色呼吸光圈
- 第二、第三推荐：较弱光圈
- 从入口到座位绘制简单路径
- 右侧显示推荐原因

例如：

```
推荐 B07

距离入口：18 米
连续可用：2 小时
附近占用率：35%
靠近插座
距离讨论区域较远
```

这里的路径不需要接入校园导航。管理员在布局编辑器里设置入口坐标、座位坐标和过道即可。

#### 4. 不建议实时传输真实鼠标光标

==我的建议是效果跟直播间的状态栏一样==

更合理的是做成匿名的“座位关注状态”：

- `3 人正在浏览本自习室`
- `2 人正在查看 B07`
- `B07 正在被其他用户临时锁定`
- 不显示姓名，不展示真实鼠标轨迹
- 只在点击或悬停某座位一定时间后更新

## 二、临时锁座与倒计时：最适合展示分布式特色

==跟腾讯在线文档多人编辑差不多，但是我觉得意义不大==

目前用户点击座位后，很可能直到提交预约才发生后端竞争。可以增加一个类似购票系统的“临时锁座”。

### 交互流程

1. 用户点击 B07。
2. B07 立即变成黄色。
3. 座位上显示倒计时：`01:29`。
4. 其他用户实时看到“正在选择中”。
5. 用户确认预约后转为正式预约。
6. 倒计时结束仍未提交，自动释放。

界面效果：

```
B07
其他用户正在选择
剩余 47 秒
```

### 技术价值

可以使用：

- Redis 保存临时锁，设置 TTL
- Redisson 保证同一座位同一时间段的并发操作
- SSE 推送临时锁定和释放事件
- MySQL 唯一索引做最终兜底

这能把你们已有的分布式架构真正展示出来，而不仅仅是在答辩 PPT 中说“使用了 Redis 分布式锁”。

### 建议事件类型

```
SEAT_HOLD_CREATED
SEAT_HOLD_EXPIRED
RESERVATION_CREATED
RESERVATION_CANCELLED
CHECK_IN_COMPLETED
SEAT_RELEASED
```

每次事件发生时，座位状态用短动画过渡，而不是整页刷新。

## 三、可解释的多目标推荐

目前“附近空位推荐”主要是按照距离和空位数推荐自习室。可以升级为座位级、多目标、可解释推荐。

### 1. 用户可以选择偏好

预约前提供几个轻量选项：

```
优先连续可用
优先距离入口近
优先安静
优先靠近插座
优先靠窗
优先人少
```

不必做复杂机器学习，可以采用权重评分：

```
综合分 =
距离得分 × 0.20
+ 连续可用得分 × 0.30
+ 偏好匹配得分 × 0.25
+ 周边拥挤度得分 × 0.15
+ 房间利用均衡得分 × 0.10
```

管理员可以给座位设置属性：

- 靠窗
- 有插座
- 靠近入口
- 安静区域
- 讨论区域
- 无障碍座位
- 空调直吹区域

重点不在公式多复杂，而在于系统能给出解释：

> 推荐 A05，因为它在目标时间内连续空闲 150 分钟，靠近插座，周围座位占用率较低。

不要轻易把这种权重算法称为“AI”。答辩时称为“可解释的多目标座位推荐”反而更严谨。

------

### 2. 冲突后自动提供替代方案

用户点击提交时，如果座位刚好被别人抢走，不能只弹出：

```
预约失败，座位已被占用
```

应该立即给出：

```
B07 已被预约，为你找到以下替代方案：

B06：相邻座位，同一房间
C07：距离 2 米，连续可用 2 小时
A302-B05：同楼层，空位较多
```

用户可以一键切换并提交，而不必重新返回筛选页面。

这会明显提升系统的“智能感”。

## 四、候补队列与座位接力

==跟12306抢票差不多==

当一个自习室没有空位时，不应该只有“无座位”。

用户可以加入候补队列：

```
候补条件：
时间：14:00—16:00
区域：A301 或同楼层
偏好：有插座
接受替代座位：是
```

座位因取消、提前结束、超时释放而空出时：

1. 系统查找符合条件的候补用户。
2. 按进入队列时间、积分或公平策略排序。
3. 给第一位用户保留 30 秒。
4. 用户接受后转为正式预约。
5. 超时未接受则转给下一位。

用户看到：

```
你当前排在第 2 位
预计等待时间：约 12 分钟
```

预计等待时间可以根据历史取消和提前释放数据估算，不要求非常准确。

这个功能的亮点在于它把：

- 超时释放
- 主动取消
- 实时推送
- 推荐算法
- 临时锁座

串成了一个完整业务闭环。

## 六、组队预约相邻座位

对于讨论自习室，可以增加“结伴学习模式”。

### 用户流程

1. 发起人选择人数，例如 3 人。
2. 系统推荐相邻座位组合。
3. 生成邀请码或组队码。
4. 成员确认后统一完成预约。

推荐结果：

```
推荐组合一：B04、B05、B06
三座连续，靠近讨论区

推荐组合二：C04、C05、D04
距离较近，连续可用 2 小时
```

### 技术难点也是亮点

必须保证：

- 三个座位全部成功，才算预约成功
- 不能只成功一部分
- 临时锁必须同时占用多个座位
- 有一个座位冲突时，整体回滚并推荐下一组

这非常适合展示分布式并发控制，但实现复杂度高于单座位预约，可以作为加分功能。

## 七、管理员端：从“表单管理”升级为“空间编辑器”

你指出“不能添加自习室、不能自定义行列”是基础缺陷。但只是增加行数和列数输入框仍然不够亮眼。

建议做成真正的可视化布局编辑器。

### 1. 拖拽式布局

管理员可以从左侧组件栏拖入：

- 普通座位
- 双人桌
- 过道
- 门
- 窗户
- 插座
- 柱子
- 讲台
- 禁用区域

支持：

- 框选多个座位
- 批量编号
- 批量启用或禁用
- 行列对齐
- 复制一整排
- 设置座位属性
- 设置入口位置

“固定行列生成”适合作为快捷模板，而拖拽编辑负责不规则房间。

------

### 2. 布局版本管理

不要直接覆盖当前布局，可以有：

```
当前版本：A301-v3
启用时间：2026-07-08

草稿版本：A301-v4
修改内容：增加 6 个座位，禁用 A07
```

管理员可以：

- 保存草稿
- 预览学生端效果
- 指定生效时间
- 回滚历史版本

这样可以避免修改布局后影响已经存在的预约。

纯粹的布局编辑是基础功能，“布局版本 + 预约兼容”才是更完整的设计亮点。

------

## 八、管理端实时看板与历史回放

你们目前管理员需要先选择时间才能看座位状态，这确实不符合实时看板的定位。

### 1. 默认进入“现在”模式

进入页面立即显示：

```
当前时间：15:42:18
空闲：41
临时锁定：2
待签到：3
使用中：19
即将释放：4
```

时间筛选放在二级功能中：

```
现在 | 未来时段 | 历史回放
```

------

### 2. 实时事件流

右侧增加事件列表：

```
15:42:18  B07 完成签到
15:41:53  A05 临时锁定
15:41:21  C03 超时释放
15:40:58  D04 主动结束使用
```

点击事件后，对应座位闪烁定位。

这比单纯变化数字更能体现“系统正在实时工作”。

------

### 3. 历史状态回放

管理员选择某个时间段，拖动播放条：

```
14:00  ▶ ━━━━━●━━━━ 16:00
```

座位图根据历史事件逐步变化，可以回答：

- 哪个时刻最拥挤？
- 哪些区域一直无人使用？
- 哪些座位频繁被取消？
- 某次异常占用是如何发生的？

不必引入完整的事件溯源架构，只需要保存关键座位状态事件，即可重建简化历史状态。

这是非常适合答辩演示的管理员亮点。

## 十、设施问题与座位反馈闭环

可以让学生对座位提交快速反馈：

```
插座损坏
桌面损坏
灯光异常
附近噪声较大
座椅损坏
```

管理员确认后：

- 临时禁用座位
- 添加维修中状态
- 影响推荐排序
- 修复后重新启用

座位图上显示：

```
A07
暂不可用
原因：插座维修
预计恢复：明日 10:00
```

这不依赖门禁或物联网，但让系统从“预约工具”发展成“自习空间管理平台”。

需要注意防止恶意举报，可以要求同一用户短时间内不能重复举报，并由管理员确认后才正式禁用。

## 十一、站内通知中心

不需要接短信、微信或校园系统，只做站内通知即可：

- 预约即将开始
- 签到剩余时间不足
- 候补座位已为你保留
- 座位被管理员调整
- 积分增加或扣除
- 黑名单即将解除
- 提前释放成功

通知内容不能只写“积分发生变化”，而应写清楚原因：

```
积分 +2
原因：按时签到并完成本次预约

积分 -3
原因：预约开始后 15 分钟内未签到
```

积分变化事件可以通过 SSE 同步更新顶部积分、积分流水和排行榜，避免三个位置状态不一致。
